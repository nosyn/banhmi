# CSRF

`CsrfModule` protects against Cross-Site Request Forgery attacks using the
double-submit cookie pattern with tokens generated by `@banhmi/crypto`.

## When to use

When your API is consumed by a browser-based SPA or a server-rendered form,
add `CsrfModule.forRoot()` to prevent CSRF attacks on state-mutating endpoints.

## Setup

```bash
bun add @banhmi/security
```

## How it works

1. On **safe** methods (`GET`, `HEAD`, `OPTIONS`) the middleware issues a
   `csrf-token` cookie (if absent) and reflects the value in the
   `x-csrf-token` response header so SPA clients can read it.
2. On **unsafe** methods (`POST`, `PUT`, `PATCH`, `DELETE`) the middleware
   compares the `csrf-token` cookie with the `x-csrf-token` request header
   (or `_csrf` form field). Mismatch → `403 { message: 'CSRF token mismatch' }`.

## Configuration

```ts
CsrfModule.forRoot({
  cookieName: 'csrf-token',       // default
  headerName: 'x-csrf-token',     // default
  formField: '_csrf',             // default
  cookie: { secure: true, sameSite: 'lax' },
})
```

## API reference

### `CsrfModule.forRoot(opts?)`

`opts` is a `CsrfOptions` object. All keys are optional.
