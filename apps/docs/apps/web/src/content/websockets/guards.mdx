# Guards

Guards work identically in WebSocket gateways as in HTTP controllers. They
receive a `WsContext` execution context and decide whether the message handler
should be invoked.

## When to use

- Reject unauthenticated WebSocket connections before the handler runs.
- Enforce per-message permissions in a gateway.

## Usage

```ts
import { CanActivate, ExecutionContext, Injectable } from 'banhmi'

@Injectable()
export class WsAuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const client = context.switchToWs().getClient()
    const token = client.data?.token as string | undefined
    return verifyJwt(token)
  }
}
```

Apply at gateway class or per-message-handler level:

```ts
@UseGuards(WsAuthGuard)
@WebSocketGateway({ path: '/ws' })
export class ChatGateway {}
```

## Connection-phase guard

To reject a WebSocket during the initial HTTP upgrade (before the socket opens),
implement `handleConnection` and throw `WsException`:

```ts
handleConnection(client: BunWebSocket) {
  if (!verifyJwt(client.data?.token)) {
    client.close(4001, 'Unauthorized')
  }
}
```

## API reference

- `CanActivate` — same interface as HTTP guards; switch with `ctx.switchToWs()`.
- `ctx.switchToWs().getClient()` — the raw `BunWebSocket` instance.
- `ctx.switchToWs().getData()` — the parsed message payload.
- `WsException(message)` — the WebSocket-specific exception class.
