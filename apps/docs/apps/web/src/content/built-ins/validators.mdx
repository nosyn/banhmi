# Built-in Validators

`@banhmi/validation` provides adapter-based validation with zero required
external dependencies. The class-validator-style decorators let you annotate
DTOs declaratively using TC39 Stage 3 decorators and `Symbol.metadata`.

## Class-validator-style decorators

Import from `@banhmi/validation` (or the `banhmi` meta-package):

```ts
import { IsString, IsEmail, classValidator, AdaptedValidationPipe } from '@banhmi/validation'
```

| Decorator | Parameters | Description |
|---|---|---|
| `@IsOptional()` | `opts?` | Skip all other rules when value is `undefined` or `null`. |

| Decorator | Parameters | Description |
|---|---|---|
| `@IsDate()` | `opts?` | Value must be a valid `Date` instance. |

### Object / nested decorators

| Decorator | Parameters | Description |
|---|---|---|
| `@IsObject()` | `opts?` | Value must be a plain object (not `null`, not array). |
| `@ValidateNested(typeFn)` | `typeFn: () => unknown, opts?` | Recursively validate nested DTO. Accepts single objects or arrays. |

All `opts?` parameters accept `{ message?: string }` to override the default
error message.

## `classValidator(Cls)` builder

Builds a `Validator<T>` from a decorated class. Compatible with
`AdaptedValidationPipe`:

```ts
class CreateCatDto {
  @IsString() @MinLength(2) name!: string
  @IsInt() @Min(0)          age!: number
}

const pipe = new AdaptedValidationPipe(classValidator(CreateCatDto))
```

On failure the pipe throws `ValidationException` (a `BadRequestException`)
whose `errors` field is a `ValidationError[]` array with `path` and `message`.

## Error format

```json
{
  "message": "Validation failed",
  "errors": [
    { "path": ["name"], "message": "length must be at least 2" },
    { "path": ["age"],  "message": "must be at least 0" }
  ]
}
```
