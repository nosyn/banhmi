# Lazy-loading Modules

Lazy-loading defers module initialisation until the first request that needs
it. This reduces cold-start time for large applications where many code paths
are rarely used.

## When to use

- Serverless / edge functions where cold-start latency matters.
- Large monolithic APIs where only a subset of routes are exercised in any
  given deployment.
- CLI-adjacent code that mounts optional feature modules on demand.

## Usage

```ts
import { LazyModuleLoader } from 'banhmi'

@Injectable()
export class CatsService {
  static inject = [LazyModuleLoader] as const
  constructor(private lazyLoader: LazyModuleLoader) {}

  async getHeavyFeature() {
    const { HeavyModule } = await import('./heavy/heavy.module')
    const moduleRef = await this.lazyLoader.load(() => HeavyModule)
    const service = moduleRef.get(HeavyService)
    return service.compute()
  }
}
```

## Caveats

- Lazy-loaded modules do **not** run lifecycle hooks on the boot path.
- Guards, interceptors, and filters declared inside lazy modules only apply
  after the first load.
- Avoid lazy-loading modules that have side effects expected at startup.

## API reference

- `LazyModuleLoader` — injectable service; provided automatically by Banhmi core.
- `lazyLoader.load(() => ModuleClass)` — resolves and returns a `ModuleRef`.
- `ModuleRef.get(Token)` — retrieve a provider from the lazily loaded module.
