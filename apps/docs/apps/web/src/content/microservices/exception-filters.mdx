# Exception Filters

Microservice exception filters catch errors thrown by `@MessagePattern` handlers
and return a serialised `MicroserviceResponse` with an `error` field.

## When to use

- You want structured error responses instead of unhandled transport errors.
- You need to catch domain-specific exceptions and transform them.

## Default behaviour

If no filter is registered, `DefaultMsExceptionFilter` is used automatically.
It serialises the thrown error as `{ error: { message, status } }` where:
- `status` is taken from `HttpException.statusCode` or `err.status`, defaulting to `500`.

## Defining a custom filter

```ts
import type { MsExceptionFilter, MsExecutionContext, MicroserviceResponse } from '@banhmi/microservices'

class DomainErrorFilter implements MsExceptionFilter<DomainError> {
  catch(err: DomainError, _ctx: MsExecutionContext): MicroserviceResponse {
    return { error: { message: err.message, status: 422, code: err.code } }
  }
}
```

## Using the enhancer pipeline

```ts
import { runMsEnhancerPipeline } from '@banhmi/microservices'

const result = await runMsEnhancerPipeline(
  ctx,
  () => handler(msg),   // the bound handler
  [],                    // guards
  [],                    // interceptors
  [{ filterInstance: new DomainErrorFilter(), type: DomainError }],
)
```

## API reference

- `runMsEnhancerPipeline(ctx, handler, guards, interceptors, filters)` — runs the pipeline.
- `MsExceptionFilter<T>` — `catch(exception: T, ctx: MsExecutionContext): MicroserviceResponse`.
- `DefaultMsExceptionFilter` — built-in filter that serialises all errors.
- `MsExecutionContext` — `getMessage()`, `getClass()`, `getHandler()`.
