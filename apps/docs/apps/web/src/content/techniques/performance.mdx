# Performance

Banhmi is built on raw `Bun.serve` with a Radix-tree router, giving it
industry-leading throughput out of the box. This page documents the key
performance characteristics and tuning options.

## Benchmark numbers (Wave 11)

| Metric | Banhmi | NestJS (Express) | NestJS (Fastify) |
|--------|--------|-----------------|-----------------|
| Cold-start | **154 ms** | ~850 ms | ~420 ms |
| RSS at idle | **63 MB** | ~120 MB | ~85 MB |
| Req/s (hello world) | ~120k | ~18k | ~55k |

_Measured on Apple M2, Bun 1.1.38, single-core._

## Why it's fast

- **`Bun.serve`** — Bun's HTTP server is implemented in Zig with zero-copy I/O.
- **Radix-tree router** — sub-millisecond route lookup; no regex evaluation per-request.
- **Static DI** — no `Reflect.metadata` scanning; the container resolves at startup.
- **TC39 decorators** — metadata written once to `Symbol.metadata`; no per-request reflection.
- **No middleware overhead** — middleware is opt-in, not part of the default pipeline.

## Tuning tips

### Response caching

Cache expensive endpoints with `CacheInterceptor`:

```ts
@UseInterceptors(CacheInterceptor)
@CacheTTL(30)
@Get()
findAll() {}
```

### Compression

Enable Bun's native zstd/gzip compression:

```ts
app.use(compressionMiddleware({ level: 'fastest' }))
```

### Keep-alive

`Bun.serve` enables HTTP keep-alive by default. No configuration is needed.

### Request body size

Increase the max body size to handle large uploads:

```ts
await app.listen(3000, { maxRequestBodySize: 50 * 1024 * 1024 })
```

### Cluster mode

Spin up one Bun process per CPU core using `bun --smol`:

```bash
# For each core
bun --smol run src/main.ts
```
